# 修理進捗管理システム

修理案件の受付、担当者割り当て、訪問予定の共有、完了報告、履歴管理までを一元化する Web アプリケーションです。  
バックエンドのデータベースには Google スプレッドシートを利用し、現場運用で扱いやすい表形式のデータと Web UI を組み合わせています。

このリポジトリはポートフォリオ公開用です。画面上に表示されるサンプルはすべて架空のテストデータであり、実在の顧客名、住所、電話番号、メールアドレス、企業固有の情報は含めていません。

## 目的

修理対応の現場では、受付済みの依頼、電話確認中の案件、担当者への依頼状況、訪問予定、完了報告が複数の連絡手段や表に分散しやすくなります。

このシステムでは、管理者と現場担当者の情報共有をシンプルにするため、次の状態をひとつの画面で追えるようにしています。

- どの案件が未対応か
- どの案件が電話確認中か
- どの担当者へ依頼済みか
- どの案件が対応中か
- 訪問予定日時が登録されているか
- 完了済みの案件が履歴に残っているか

## システム概要

管理者は `/admin` から案件一覧を確認し、ステータス更新や担当者割り当てを行います。担当者は案件ごとの報告ページ `/report/[id]` から、訪問予定日時や修理完了報告を登録します。

データは Google スプレッドシートに保存されます。アプリケーションは Google Sheets API を通じて案件と担当者情報を読み書きします。環境変数が未設定の場合は、アプリ内のフォールバック用テストデータで動作確認できます。

## 主な機能

- 管理者ダッシュボードで未完了案件を一覧表示
- 案件ステータスの更新
- 担当者の割り当て、担当者変更
- 担当者ごとの依頼中・対応中件数の表示
- 担当エリアに基づく担当者推薦
- 店舗名、店舗担当者、住所によるキーワード検索
- 受付日ベースの日付絞り込み
- 現場担当者向けの訪問予定登録
- 現場担当者向けの修理完了報告
- 完了案件の履歴・ログ表示
- 完了案件を未完了状態へ戻す操作
- 案件作成、編集、完全削除
- Google Drive フォルダ配下の複数スプレッドシート読み込み
- Google Sheets API によるデータ永続化
- 認証情報未設定時のテストデータ表示

## 想定ワークフロー

1. 管理者が修理案件を確認する
2. 必要に応じて電話確認中へステータスを変更する
3. エリアや稼働状況を見ながら担当者を割り当てる
4. 割り当て後、案件ステータスは依頼中になる
5. 担当者が報告ページから訪問予定日時を登録する
6. 訪問予定登録後、案件ステータスは対応中になる
7. 作業完了後、担当者が修理完了報告を行う
8. 完了案件は履歴・ログ画面に移動する

## ステータス

| ステータス | 意味 |
| --- | --- |
| 未対応 | 受付後、まだ確認や依頼が始まっていない状態 |
| 電話確認中 | 管理者が状況確認を進めている状態 |
| 依頼中 | 担当者へ作業依頼済みで、訪問予定の登録待ちの状態 |
| 対応中 | 訪問予定が登録され、作業対応中の状態 |
| 修理完了 | 現地作業が完了した状態 |
| 口頭完了 | 電話や口頭案内などで完了した状態 |

## データベース

このシステムでは、データベースとして Google スプレッドシートを利用しています。

通常の RDB ではなくスプレッドシートを選んだ理由は、非エンジニアでもデータを直接確認・編集しやすく、既存業務の表管理から Web アプリへ移行しやすいためです。アプリ側では Google Sheets API を使って、案件情報と担当者情報を参照・更新します。

### 利用するシート

- `cases` : 修理案件データ
- `technicians` : 担当者データ

案件データと担当者データは、同一スプレッドシートでも別々のスプレッドシートでも運用できます。

## データ取得と更新

### 案件一覧

`getCasesData()` が `cases` シートから A-R 列を取得し、アプリ用の `RepairCase` 型へ変換します。Google Drive フォルダ ID が設定されている場合は、フォルダ配下の複数スプレッドシートから案件を集約できます。

### 担当者一覧

`getTechniciansData()` が `technicians` シートから A-G 列を取得し、担当者名、会社名、対応エリア、連絡先を画面に表示します。

### 更新処理

Server Actions から Google Sheets API を呼び出し、対象行を更新します。

- ステータス更新: `cases` シート P 列
- 担当者割り当て: `cases` シート Q 列
- 訪問予定日時: `cases` シート R 列
- 訪問予定の履歴追記: `cases` シート N 列
- 案件編集: `cases` シート D-M 列
- 案件削除: 対象行の削除

## 担当者推薦

担当者割り当てでは、案件の住所や店舗名と担当者の対応エリアを照合し、条件に合う担当者を推奨表示します。

また、サイドバーでは担当者ごとに現在の依頼中・対応中の件数を表示します。これにより、エリアだけでなく稼働状況を見ながら割り当てを判断できます。

## テストデータ

環境変数が未設定の場合、アプリ内のテストデータを使って画面を確認できます。

現在のフォールバックデータ:

- 未完了案件: 6 件
- 履歴・ログ用の完了案件: 2 件
- 担当者: 6 名

テストデータには以下の状態が含まれます。

- 未対応
- 電話確認中
- 依頼中
- 対応中
- 修理完了
- 口頭完了

Google Sheets に同等のテストデータを投入する場合は、次のコマンドを実行します。

```bash
npm run seed
```

`scripts/seed.js` は `cases` シートと `technicians` シートへテストデータを追記します。既存データは削除しません。

## ローカル実行

```bash
npm install
npm run dev
```

ブラウザで `http://localhost:3000` を開きます。

Google Sheets の認証情報がない場合でも、テストデータで以下を確認できます。

- `http://localhost:3000/admin`
- `http://localhost:3000/admin/history`
- `http://localhost:3000/report/1`

## 環境変数

`.env.example` を参考に `.env.local` を作成します。

```bash
cp .env.example .env.local
```

| 変数名 | 用途 |
| --- | --- |
| `GOOGLE_SHEET_ID` | 案件と担当者を同じスプレッドシートで管理する場合の ID |
| `GOOGLE_SHEET_ID_CASES` | 案件管理用スプレッドシート ID |
| `GOOGLE_SHEET_ID_TECHNICIANS` | 担当者管理用スプレッドシート ID |
| `GOOGLE_DRIVE_FOLDER_ID_CASES` | 複数の案件スプレッドシートを集約する Google Drive フォルダ ID |
| `GOOGLE_CLIENT_EMAIL` | Google サービスアカウントのメールアドレス |
| `GOOGLE_PRIVATE_KEY` | Google サービスアカウントの秘密鍵 |

## Google Cloud / Sheets 側の準備

1. Google Cloud でプロジェクトを作成する
2. Google Sheets API を有効化する
3. 必要に応じて Google Drive API を有効化する
4. サービスアカウントを作成する
5. サービスアカウントのメールアドレスを対象スプレッドシートへ共有する
6. `.env.local` にスプレッドシート ID とサービスアカウント認証情報を設定する

本番運用では、秘密鍵をリポジトリに含めず、デプロイ先の環境変数として登録します。

## デプロイ

Dockerfile と Cloud Run 向けの設定を含めています。デプロイ時は `.env.local` をアップロードせず、ホスティング環境の環境変数に値を設定します。

## セキュリティと公開時の注意

- `.env.local` はコミットしない
- サービスアカウントの JSON キーをコミットしない
- Google Sheets の共有権限は必要最小限にする
- 公開デモには実顧客名、実住所、実電話番号、実メールアドレスを含めない
- Google Cloud の秘密鍵は漏洩時に即時削除・再発行する
- GitHub の secret scanning alerts を確認する
